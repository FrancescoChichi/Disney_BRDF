Fundamentals of Computer Graphics – Final Projects Proposals

In this project you will expand on our free, and tiny, path tracer to either add new lighting effects, import complex models, or improve the underlying system implementation.

The main difference between the projects and the homeworks is that we want the projects to not be guided by us to see how well you can move in a new topic and how well you can work independently. So compared to the homeworks, the projects are harder.

Remember though that the project is not required to pass the class, so only choose it if you feel comfortable with the topic and you want to explore graphics more. If you do well in the projects, you might even continue down the line for a full thesis in graphics.

Remember also that you can work on groups of up to two people. The limits on the number of people is to keep things fair with respect to people that prefer to work alone.

Project Modalities

  • Choose a topic from the list below.
  • By the end of the lectures, please communicate whether you will try to do the project or prefer to finish with the homeworks using the form provided.
  • You can hand in the project at any future "appello" up to and including the September 2018 one.
  • Please be sure that the professor has a copy of the project materials 5 days before the exam so the project can be graded; submission will happen on submission links published on the Google Groups.
  • For each project you have to hand in
    • a PDF detailing the features implemented, scenes, rendering times and parameters
    • all your code that should either be modification of the libraries given or be contained in up to two files (.h/.cpp) pair
    • input scenes in a format that the system can freely load
    • at least one, impressive, output image rendered at high resolution and with no noise
    • supporting images to show the various features
  • Please consider that we want to release your code and/or data as part of our library (obviously with written acknowledgments to you and only if you allows us). So in this case, I will also evaluate how well your code is written and whether it is worthy of release.

Policies

You have to start and modify the Yocto/GL version that will be release the last day of class. Since this is a heavily modified version, it will contain bugs and compilation problems that we cannot fix in time for class end. For small changes, please submit Issues with GitHub especially if you have a fix for it.

You can use any open source code that you have to rights to and that can be integrated in Yocto/GL. Practically this means that the code you use and/or integrate as to be release under either the MIT, BSD or Apoache license; but not the GPL. You have to include the license of the code you use, even if you simply take inspiration from it, inside the files you give back to me. For 3D models, you can use any mode released under the Creative Common license. Please include the license file with the data.

In particular, the main open source renderer avialable to you is pbrt that also has a full book on it. You can use any code from it, but really you should heavily adapt it to Yocto/GL whose design is the opposite of pbrt, namely the code should stay very small and opinionated in not supporting all features but the coolest ones.

If you use unauthorized code, or do not give proper credit to the work of others, I will promptly fail you from the class since this is a major copyright violation with legal implications; if in doubt ask the professor.

If you work in two, either choose a "hard" project or significantly improve the quality of the produced images or the implementation.

For projects it is not easy to say when it is "good enough" since one can always improve more the work; to make things easier for you and make sure that your final grade matches your expectation you can ask me to quickly review your submission by sending me one representative high-quality image and your code specifying exactly where to look. You can ask me appointments to discuss the projects which can happen in person or Skype, but please consider that I have limited bandwidth and cannot review your work constantly, nor I want to since that would not show your independence.

It would make life easier if you work on a public Git repo so I can grab you latest stuff easily – include images there (I know it is not good, but hey gotta go easy way here). For final submission, I will open a link to a submission form roughly 10 days before any appello.

Project Ideas

I attempted to include different project ideas that vary from mostly system work to mostly math-y stuff. You are welcome to choose anything that you want. The rating is my guess and may be completely wrong depending on your background, strengths and inclinations. For me, the order in easy to hard, less fun to more fun. But that's just me.

  1. [Easy] Small scenes 1. Port most scenes from here to Yocto/GL. You can skip a couple if they really do not render, but they mostly should. All scenes should be save in OBJ and glTF, this means you likely have to write a parser. The images rendered in Yocto should match the once rendered in pbrt or another of the supported renderers. This means that you should implement proper handling for glass and mirrors (deltas)

  2. [Easy] Small scenes 2. Port most of the small scenes from here to Yocto/GL. You can skip a couple if they really do not render, but they mostly should.

  3. [Easy] Large ecosystem scene. Port both nature ecosystems scenes from pbrt in a way that all images match with that renderer. See above for notes (but no glass here). You code have to absolutely support instancing otherwise it is bad.

  4. [Easy] Properly integrate embree in Yocto. Optionally switch intersection code inside the scene functions. The renderer should not know about this. Compiles and links on OsX, Windows and Linux.

  5. [Easy] Implement rough reflection and transmission from here with sampling from here.

  6. [Medium] Implement the Disney BRDF from here. Please do not include the true subsurface scattering but only the fake term.

  7. [Medium] Hair rendering with approximate scattering from here with data from here.

  8. [Hard] Implement proper normal mapping from here.

  9. [Hard] Correct hair rendering from pbrt with same data as above.

  10. [Hard] Skin rendering with BSSRDF. Implement subsurface scattering with the BSSRDF presented in here.

  11. [Hard] Skin rendering with volumes. Implement the skin technique in here
